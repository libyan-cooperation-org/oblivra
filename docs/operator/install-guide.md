# OBLIVRA Installation Guide

This guide walks an operator through installing OBLIVRA on Linux,
Windows, or macOS — both as a standalone single-node deployment and
as a 3-node Raft cluster — with explicit air-gap-friendly steps for
sovereign deployments.

**Audience**: DevOps / SRE / Security engineers responsible for
deploying and operating the OBLIVRA platform. Assumes familiarity
with the host operating system's service manager (systemd, launchd,
Windows SCM) and basic TLS / certificate concepts.

## 1. System Requirements

### Minimum (single-node, < 5,000 EPS)

| Component | Requirement |
|---|---|
| **OS** | Linux kernel 5.10+, Windows Server 2019+, macOS 12+ |
| **CPU** | 2 cores (x86_64 or arm64) |
| **RAM** | 4 GB |
| **Disk** | 50 GB SSD (hot tier) + 100 GB HDD (warm + cold) |
| **Network** | Outbound 443/tcp to update channel; agents reach the server on configurable port (default 8443) |
| **Open file descriptors** | 32,768 (`ulimit -n`) |

### Recommended (production single-node, 5,000–50,000 EPS)

| Component | Requirement |
|---|---|
| **CPU** | 8 cores |
| **RAM** | 32 GB |
| **Disk** | 500 GB NVMe (hot) + 1 TB SSD (warm) + S3-compatible cold tier |
| **Network** | 1 Gbps |
| **OS** | Linux only (eBPF-based agent telemetry requires kernel 5.10+) |

### Production cluster (3-node Raft, 50,000+ EPS)

Each node should match the recommended single-node specs above.
Network: <2 ms RTT between cluster nodes is required for healthy
Raft elections.

### Air-gap deployment

Same hardware, **plus**:
- Offline update bundle delivery channel (USB, signed-CD, internal mirror)
- No outbound internet — every dependency ships in the install bundle
- Optional: TPM 2.0 / Secure Enclave for hardware root of trust

## 2. Pre-flight Checklist

Before installing, confirm:

- [ ] **Time sync is configured** — chrony / systemd-timesyncd / w32time. Audit-chain integrity depends on monotonic timestamps; >5-minute clock skew triggers alerts.
- [ ] **Hostname is stable** — `/etc/hostname` reflects the operator-meaningful name. Hostnames are baked into the agent's stable AgentID derivation.
- [ ] **Firewall plan** — decide which ports are reachable from where:
  - **8443/tcp** — REST + UI (HTTPS, operator workstation)
  - **15300/tcp** — Raft inter-node (cluster nodes only)
  - **4433/tcp** — agent ingest (every monitored host)
  - **9090/tcp** — Prometheus metrics scrape (monitoring system only)
- [ ] **Firewall outbound** — agents need to reach the server. Operator workstations need to reach the UI. **No other outbound traffic is required** in air-gap mode.
- [ ] **TLS certificates** — either operator-supplied (recommended for production) or auto-generated by `cmd/certgen` on first boot. For agents to validate the server, agents must trust the server's CA (see §6).
- [ ] **Disk layout** — separate filesystem for `<DataDir>/badger/` (hot tier) and `<DataDir>/archives/` (warm tier) so a hot-tier full-up doesn't block the warm tier from accepting promotions.
- [ ] **Backup target** — destination for nightly snapshots. See [Backup & Restore](#backup--restore) below.
- [ ] **License key** (if you have one) — `OBL-XXXX-XXXX-XXXX` format. Required for premium features (UEBA, FIDO2 quorum, multi-tenant). Free tier covers single-tenant + ≤5 agents.

## 3. Single-Node Install

### 3.1 Linux (systemd)

#### Download + verify

```bash
# 1. Download the release tarball + signature
curl -LO https://releases.oblivra.io/v1.5.0/oblivra-v1.5.0-linux-amd64.tar.gz
curl -LO https://releases.oblivra.io/v1.5.0/oblivra-v1.5.0-linux-amd64.tar.gz.sig

# 2. Verify the signature against our release-signing public key
curl -L https://oblivra.io/release-signing-key.asc | gpg --import
gpg --verify oblivra-v1.5.0-linux-amd64.tar.gz.sig oblivra-v1.5.0-linux-amd64.tar.gz
# Expected output: "Good signature from OBLIVRA Release Signing"
```

If the signature fails, **stop**. Do not run an unverified binary.

#### Install

```bash
# 3. Extract to /opt/oblivra
sudo mkdir -p /opt/oblivra
sudo tar -xzf oblivra-v1.5.0-linux-amd64.tar.gz -C /opt/oblivra --strip-components=1

# 4. Create the dedicated user
sudo useradd --system --no-create-home --shell /usr/sbin/nologin oblivra

# 5. Create data + log directories
sudo mkdir -p /var/lib/oblivra /var/log/oblivra
sudo chown -R oblivra:oblivra /var/lib/oblivra /var/log/oblivra
sudo chmod 700 /var/lib/oblivra        # data is sensitive — owner-only

# 6. Install the systemd unit
sudo cp /opt/oblivra/contrib/systemd/oblivra.service /etc/systemd/system/
sudo systemctl daemon-reload
```

#### systemd unit — `oblivra.service`

The shipped unit (`/opt/oblivra/contrib/systemd/oblivra.service`) looks like:

```ini
[Unit]
Description=OBLIVRA Sovereign SIEM
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=oblivra
Group=oblivra
ExecStart=/opt/oblivra/bin/oblivra-server \
    --data-dir=/var/lib/oblivra \
    --config=/etc/oblivra/server.toml
Restart=always
RestartSec=5s

# Watchdog auto-restart contract: when the agent / server exits with
# code 75 (RestartExitCode), systemd treats it as a "transient
# failure, restart me" signal. See internal/agent/restart.go.
SuccessExitStatus=0 75

# Hardening
NoNewPrivileges=true
ProtectSystem=strict
ProtectHome=true
PrivateTmp=true
ReadWritePaths=/var/lib/oblivra /var/log/oblivra
CapabilityBoundingSet=
AmbientCapabilities=

# Resource limits
LimitNOFILE=65536
LimitNPROC=4096

[Install]
WantedBy=multi-user.target
```

#### Start

```bash
sudo systemctl enable --now oblivra
sudo systemctl status oblivra
```

#### Agent install (each monitored host)

```bash
# 1. Same download/verify pattern, agent binary instead.
curl -LO https://releases.oblivra.io/v1.5.0/oblivra-agent-v1.5.0-linux-amd64.tar.gz
sudo tar -xzf oblivra-agent-v1.5.0-linux-amd64.tar.gz -C /opt/oblivra-agent --strip-components=1

# 2. Install the systemd unit + initial config
sudo cp /opt/oblivra-agent/contrib/systemd/oblivra-agent.service /etc/systemd/system/

sudo mkdir -p /etc/oblivra-agent
sudo tee /etc/oblivra-agent/config.toml > /dev/null <<EOF
server = "https://siem.example.com:4433"
tenant_id = "GLOBAL"
data_dir = "/var/lib/oblivra-agent"
tls_ca = "/etc/oblivra-agent/server-ca.pem"
EOF

sudo cp <your-server-ca-cert>.pem /etc/oblivra-agent/server-ca.pem

# 3. Start
sudo systemctl enable --now oblivra-agent
sudo journalctl -u oblivra-agent -f
```

The agent's first run runs **historical backfill** — it scans
`journalctl --since "30 days ago"` plus `/var/log/{syslog,auth.log,
secure,messages,kern.log,dpkg.log}` and ships everything found.
Expect a brief CPU spike on first boot; subsequent boots are
no-ops once the `<DataDir>/backfill.complete` marker is written.

### 3.2 Windows (Service Control Manager)

```powershell
# Run as Administrator

# 1. Extract release
Expand-Archive -Path .\oblivra-v1.5.0-windows-amd64.zip -DestinationPath C:\Program Files\OBLIVRA

# 2. Verify Authenticode signature
Get-AuthenticodeSignature 'C:\Program Files\OBLIVRA\bin\oblivra-server.exe'
# Expected: Status = Valid, SignerCertificate from "OBLIVRA Release Signing"

# 3. Create the data directory with restricted ACLs
$acl = Get-Acl 'C:\Program Files\OBLIVRA'
$dataDir = 'C:\ProgramData\OBLIVRA'
New-Item -ItemType Directory -Path $dataDir -Force
icacls $dataDir /inheritance:r /grant:r "SYSTEM:(OI)(CI)F" "Administrators:(OI)(CI)F"

# 4. Install as a Windows service
sc.exe create OBLIVRA `
    binPath= "`"C:\Program Files\OBLIVRA\bin\oblivra-server.exe`" --data-dir=`"C:\ProgramData\OBLIVRA`"" `
    DisplayName= "OBLIVRA Sovereign SIEM" `
    start= auto

# 5. Configure recovery — restart on crash, with the same exit-75
# semantics as the systemd unit (Windows SCM treats any non-zero
# exit as a service failure, so we configure 3 restart attempts).
sc.exe failure OBLIVRA reset= 86400 actions= restart/5000/restart/5000/restart/5000

# 6. Start
sc.exe start OBLIVRA
```

For the agent on Windows hosts, the same pattern with
`oblivra-agent.exe`. Windows-specific note: the agent's
historical backfill uses `wevtutil.exe qe` to scan the
**System / Security / Application** event-log channels — make sure
the service account has `SeSecurityPrivilege` to read the Security
channel, otherwise that source falls back to `[skipped]` in the
backfill marker.

### 3.3 macOS (launchd)

```bash
# 1. Notarised .pkg installs to /Library/OBLIVRA/
sudo installer -pkg ./OBLIVRA-v1.5.0.pkg -target /

# 2. Verify the installer's Apple notarisation
spctl --assess --verbose=4 --type install ./OBLIVRA-v1.5.0.pkg

# 3. The installer drops a launchd plist at
#    /Library/LaunchDaemons/com.oblivra.server.plist
sudo launchctl load -w /Library/LaunchDaemons/com.oblivra.server.plist

# 4. Verify
sudo launchctl list | grep oblivra
```

The shipped plist:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
  "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
  <dict>
    <key>Label</key>          <string>com.oblivra.server</string>
    <key>Program</key>        <string>/Library/OBLIVRA/bin/oblivra-server</string>
    <key>ProgramArguments</key>
    <array>
      <string>/Library/OBLIVRA/bin/oblivra-server</string>
      <string>--data-dir=/Library/OBLIVRA/data</string>
    </array>
    <key>RunAtLoad</key>      <true/>
    <key>KeepAlive</key>      <true/>
    <key>StandardErrorPath</key> <string>/var/log/oblivra/server.err.log</string>
    <key>StandardOutPath</key>   <string>/var/log/oblivra/server.out.log</string>
    <key>UserName</key>       <string>_oblivra</string>
    <key>SoftResourceLimits</key>
    <dict><key>NumberOfFiles</key><integer>65536</integer></dict>
  </dict>
</plist>
```

`KeepAlive=true` is the macOS analogue of systemd's `Restart=always`
— launchd respawns on crash AND on the watchdog's `os.Exit(75)`.

## 4. Air-Gap Install

For sovereign / regulated environments without any outbound
internet:

### Bundle preparation (on a connected workstation)

```bash
# 1. Download the air-gap bundle (one .iso containing every dep)
curl -LO https://releases.oblivra.io/v1.5.0/oblivra-airgap-v1.5.0.iso
curl -LO https://releases.oblivra.io/v1.5.0/oblivra-airgap-v1.5.0.iso.sig

# 2. Verify
gpg --verify oblivra-airgap-v1.5.0.iso.sig oblivra-airgap-v1.5.0.iso

# 3. Burn to a signed-CD or copy to the offline transfer media
# (USB drives are acceptable for non-classified deployments)
```

### Offline install

```bash
# On the air-gapped target host:
sudo mount /dev/cdrom /mnt/oblivra-iso

# 1. Run the offline installer
sudo /mnt/oblivra-iso/install-airgap.sh

# 2. The installer drops:
#    - /opt/oblivra/                    binaries + bundled deps
#    - /etc/oblivra/                    config templates
#    - /etc/systemd/system/oblivra.service
#    - /opt/oblivra/sigma/core/         42 baseline detection rules
#    - /opt/oblivra/threatintel/offline-feeds/   30-day TI snapshot
#    - /opt/oblivra/release-signing-key.asc  for verifying future updates

# 3. There's no telemetry phone-home, no auto-update, no online
#    license check. Operator manages every aspect locally.

sudo systemctl enable --now oblivra
```

### Air-gap update channel

Operator establishes a manual signed-update path:

```bash
# Connected workstation: mirror the latest release into your update bucket
curl -LO https://releases.oblivra.io/v1.5.1/oblivra-airgap-v1.5.1.iso
gpg --verify oblivra-airgap-v1.5.1.iso.sig oblivra-airgap-v1.5.1.iso

# Burn to media, transport into the air-gap zone, then on the target:
sudo /mnt/oblivra-iso/upgrade.sh /opt/oblivra
sudo systemctl restart oblivra
```

The `upgrade.sh` script verifies the new bundle's GPG signature
against the previously-installed release-signing key, refusing to
install any binary signed by a key the operator hasn't already
trusted. This is the project's protection against a compromised
update channel — there is no automatic key rotation.

## 5. 3-Node Raft Cluster

### Topology

Three identical hosts, each running a server instance. Recommended:

```
Node A (raft-id=A, raft-bind=A:15300, api-bind=:8443)
Node B (raft-id=B, raft-bind=B:15300, api-bind=:8443)
Node C (raft-id=C, raft-bind=C:15300, api-bind=:8443)

[Operator workstation] ──HTTPS──> [LB / floating IP] ──> any node's :8443
[Agents]               ──HTTPS──> [LB / floating IP] ──> any node's :4433
```

### Bootstrap

```bash
# On Node A (the bootstrap node):
sudo -u oblivra /opt/oblivra/bin/oblivra-server \
    --data-dir=/var/lib/oblivra \
    --raft-id=A \
    --raft-bind=A.internal:15300 \
    --raft-bootstrap=true

# On Nodes B and C, join the existing cluster:
sudo -u oblivra /opt/oblivra/bin/oblivra-server \
    --data-dir=/var/lib/oblivra \
    --raft-id=B \
    --raft-bind=B.internal:15300 \
    --raft-join=A.internal:15300
```

After all three are up, edit each systemd unit to remove
`--raft-bootstrap=true` so a future restart doesn't try to
re-bootstrap.

### Cluster health verification

```bash
curl -sk https://A.internal:8443/api/v1/cluster/status | jq
# Expected: {"leader":"A","peers":["A","B","C"],"committed_index": <N>}
```

If the leader doesn't elect within 30 seconds, check inter-node
RTT (must be < 2 ms) and that both Raft (15300) and API (8443)
ports are mutually reachable.

## 6. First-Boot Configuration

The first time you open the UI (`https://<server>:8443`), OBLIVRA
runs the **Setup Wizard** — a 4-step flow that:

1. **Creates the bootstrap admin** — email + 12+ char password.
   This is one-shot: the endpoint refuses to run a second time,
   so a stolen URL can't hijack admin.
2. **Configures alert routing** — SMTP, webhook, or Slack/Discord.
3. **Selects a detection pack** — `essential` (42 rules, default),
   `extended` (200+), or `paranoid` (300+, more false positives).
4. **Orientation tour** — walks the operator through Mission Control,
   the InvestigationPanel, and the BottomDock domain selector.

Setup completion is recorded in the audit ledger as
`setup.initialized`. From this point forward, every config change
is audit-logged.

### Server config file (`/etc/oblivra/server.toml`)

```toml
[server]
bind = "0.0.0.0:8443"
agent_bind = "0.0.0.0:4433"

[tls]
cert = "/etc/oblivra/server.crt"
key  = "/etc/oblivra/server.key"
# When present, agents must present a client cert signed by this CA.
# Strongly recommended for production.
client_ca = "/etc/oblivra/agent-ca.crt"

[storage]
data_dir = "/var/lib/oblivra"

# Phase 31 storage tiering. Foundation lives in
# internal/storage/tiering. Operator tunables:
[storage.tiers]
hot_retention  = "30d"     # events older than this promote to warm
warm_retention = "150d"    # events older than (hot+warm) promote to cold
batch_size     = 10000     # max events moved per migration cycle
interval       = "1h"      # how often the migration loop runs

# Cold tier is local-disk-JSONL by default (zero-config).
# Switch to S3-compatible by uncommenting and setting credentials:
# [storage.cold.s3]
# endpoint   = "https://s3.example.com"
# bucket     = "oblivra-cold"
# region     = "us-east-1"
# access_key = "AKIA…"
# secret_key = "…"

[detection]
sigma_dir = "/opt/oblivra/sigma/core"
pack      = "essential"

[license]
# Optional. Free tier covers single-tenant + ≤5 agents.
key = ""
```

## 7. Verification

After install, run the smoke checklist:

```bash
# 1. Service is up
systemctl is-active oblivra              # → active
systemctl is-failed oblivra              # → inactive

# 2. UI responds
curl -sk https://localhost:8443/api/v1/health
# Expected: {"status":"healthy","version":"1.5.0","uptime_seconds":42}

# 3. Agent can register (run from a host that has the agent installed)
sudo journalctl -u oblivra-agent --since "5 minutes ago" | grep "registered"
# Expected: "[transport] Agent registered: id=… hostname=… acked_seq=0"

# 4. Detection rules loaded
curl -sk https://localhost:8443/api/v1/detection/rules | jq '.rules | length'
# Expected: 42 (or whatever your selected pack contains)

# 5. Storage tiering migrator running
journalctl -u oblivra | grep "tiering: cycle complete"
# Expected (within 1 hour of first boot): "tiering: cycle complete hot→warm=0 warm→cold=0 errors=0 duration=…"
```

If all five pass, the platform is functional and ready to receive
real traffic.

### Backup & Restore

OBLIVRA ships with `oblivra-cli backup` for nightly snapshots:

```bash
# Add to /etc/cron.daily/oblivra-backup
#!/bin/bash
DATE=$(date +%Y-%m-%d)
oblivra-cli backup snapshot \
    --data-dir=/var/lib/oblivra \
    --output=/backups/oblivra-snapshot-$DATE.tar.gz
```

Restore (point-in-time recovery from a snapshot):

```bash
sudo systemctl stop oblivra
sudo -u oblivra oblivra-cli backup restore \
    --input=/backups/oblivra-snapshot-2026-04-25.tar.gz \
    --data-dir=/var/lib/oblivra
sudo systemctl start oblivra
```

**Test your backups quarterly.** A snapshot you've never restored
isn't a backup, it's a hope.

## 8. Upgrade Path

### Connected (online)

```bash
# 1. Stop the service
sudo systemctl stop oblivra

# 2. Snapshot before upgrade — non-negotiable
sudo -u oblivra oblivra-cli backup snapshot \
    --data-dir=/var/lib/oblivra \
    --output=/backups/pre-upgrade-$(date +%s).tar.gz

# 3. Download + verify the new release (same pattern as install)
curl -LO https://releases.oblivra.io/v1.5.1/oblivra-v1.5.1-linux-amd64.tar.gz
gpg --verify oblivra-v1.5.1-linux-amd64.tar.gz.sig oblivra-v1.5.1-linux-amd64.tar.gz

# 4. Replace binaries (configs and data are preserved)
sudo tar -xzf oblivra-v1.5.1-linux-amd64.tar.gz -C /opt/oblivra --strip-components=1

# 5. Start
sudo systemctl start oblivra

# 6. Verify health (see §7)
```

Schema migrations run automatically on first start of the new
version. The migration runner refuses to start if the schema
version on disk is *newer* than the binary expects — it never
downgrades silently.

### Air-gap

See §4 above. The same upgrade.sh path applies.

### Rolling cluster upgrade

For a 3-node Raft cluster, upgrade one node at a time. The cluster
tolerates a single node down at a time:

```bash
# On Node B (a non-leader; check via /api/v1/cluster/status):
sudo systemctl stop oblivra
# upgrade as above
sudo systemctl start oblivra

# Wait for Node B to rejoin:
curl -sk https://B.internal:8443/api/v1/cluster/status | jq '.peers'
# Expected: ["A","B","C"]

# Then Node C, finally Node A (leader). The leader will gracefully
# step down when SIGTERM is delivered; election picks B or C.
```

## 9. Common Troubleshooting

### Agent can't connect: `x509: certificate signed by unknown authority`

The agent's CA path doesn't match the server's actual CA. Either:

```bash
# Option 1 — distribute the server's CA cert to the agent host
scp server-ca.pem agent-host:/etc/oblivra-agent/server-ca.pem
sudo systemctl restart oblivra-agent

# Option 2 — for testing only, allow insecure TLS (NEVER in production)
echo 'insecure_tls = true' | sudo tee -a /etc/oblivra-agent/config.toml
```

### Service won't start, exit code 75

That's the **expected** restart code. Check the previous run's
exit reason:

```bash
journalctl -u oblivra | grep -i "restart requested"
# "[restart] restart requested: reason=watchdog-tamper timeout=10s"
```

If the reason is `watchdog-tamper` and you didn't initiate it,
investigate immediately — a tamper attempt was detected.

### Vault is locked

The vault auto-unlocks on first boot using the key derived from
the host's TPM (Linux/Windows) or Secure Enclave (macOS). If
you've moved the binary between hosts, the key derivation fails:

```bash
# Re-initialise the vault. THIS WIPES STORED CREDENTIALS — host
# bookmarks, integration tokens, etc. SIEM events are unaffected.
sudo -u oblivra oblivra-cli vault reset --confirm-destructive
```

### "tiering: cycle errors" in the log

Check which tier failed:

```bash
journalctl -u oblivra | grep -A2 "tiering: cycle complete"
# "tiering: cycle complete hot→warm=0 warm→cold=0 errors=1 duration=2.1s"
# "tiering: warm→cold: cold write: open: permission denied …"
```

Most common cause: the cold-tier directory (`<DataDir>/cold/`)
isn't writable by the `oblivra` user. Fix:

```bash
sudo chown -R oblivra:oblivra /var/lib/oblivra/cold
sudo chmod 700 /var/lib/oblivra/cold
```

### Bundle smoke-test failed during release build

If you're building from source and `frontend/scripts/smoke-test.sh`
fails, it usually means an icon or component name was renamed
without updating the smoke test's required-strings list. Edit the
arrays in `smoke-test.sh` to match.

### Audit log claims "Merkle chain broken"

A clock skew of >5 minutes between server reboots can cause the
chain validator to flag the gap. Check `chronyc tracking` /
`w32tm /query /status`. If the skew is real and the gap is benign,
re-seal with:

```bash
sudo -u oblivra oblivra-cli audit reseal --from=<break-event-id> --reason="clock skew remediation"
```

The reseal itself is audit-logged, so a forensic auditor can see
exactly when the chain was repaired and why.

## 10. Uninstall

### Linux

```bash
sudo systemctl disable --now oblivra
sudo rm /etc/systemd/system/oblivra.service
sudo systemctl daemon-reload
sudo rm -rf /opt/oblivra /etc/oblivra

# Data — remove only if you're sure you don't need it
sudo rm -rf /var/lib/oblivra /var/log/oblivra

# User
sudo userdel oblivra
```

### Windows

```powershell
sc.exe stop OBLIVRA
sc.exe delete OBLIVRA
Remove-Item -Recurse "C:\Program Files\OBLIVRA"
# Data:
Remove-Item -Recurse "C:\ProgramData\OBLIVRA"
```

### macOS

```bash
sudo launchctl unload /Library/LaunchDaemons/com.oblivra.server.plist
sudo rm /Library/LaunchDaemons/com.oblivra.server.plist
sudo rm -rf /Library/OBLIVRA
sudo dscl . -delete /Users/_oblivra
```
